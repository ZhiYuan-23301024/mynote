## 一、先把推理侧的「世界观」立起来（1–2 周）

> 不急着写代码，先搞清楚**为什么会有这些技术**。

### 必懂的三个核心问题

你之后写项目、面试，都围着这三件事说：

1. **为什么 LLM 推理慢？**
    
    - Prefill：compute-bound（矩阵乘）
        
    - Decode：memory-bound（逐个 token，访存多）
        
    
2. **显存到底去哪了？**
    
    - 权重（fp16 / fp8）
        
    - KV cache（动态、碎片化严重）
        
    - activation / workspace
        
    
3. **为什么 batch 大了反而危险？**
    
    - decode 阶段每个请求长度不一样，长尾拖死整个 batch
        
    

### 推荐阅读（很短，但值）

- vLLM 论文：《Efficient Memory Management for Large Language Model Serving》
    
- ORCA 论文：《Orca: A Distributed Serving System for Transformer-Based Generative Models》
    
- 一篇博客就够了：
    
    👉 _“How to optimize LLM inference”_（HuggingFace 或 vLLM blog）

- Repo：`vllm-project/vllm`

### ✅ 第二阶段：理解调度 & 显存管理（2–4 周）

**目标：能讲清 vLLM 为什么快**

#### 项目 2：精读 + 小改造 vLLM Scheduler

- 重点文件：
    
    - `vllm/core/scheduler.py`
        
    - `vllm/core/block_manager.py`
        
    
- 你可以做的事（选 1 个就够）：
    
    - 打印每个 step 的 batch 构成（prefill / decode 数量）
        
    - 加一个自定义 metric（比如平均 KV block 使用率）
        
    - 手动调 `max_num_seqs`、`max_num_batched_tokens`，观察效果
        
    

📌 **这是面试高频点**：

> “vLLM 的 continuous batching 和传统 static batching 有什么区别？”

### ✅ 第三阶段：量化 & 成本优化（4–6 周）

**目标：体现你对“性价比”的理解**

#### 项目 3：FP8 / INT4 推理对比实验

- Repo：`vllm-project/vllm`（已支持 FP8 / AWQ / GPTQ）
    
- 做：
