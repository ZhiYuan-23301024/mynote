## 死锁

同时满足四个条件时，死锁发生：
- `互斥条件`：某个资源出于非共享模式
- `占有并等待`：进程已经持有资源，申请的新资源被别的进程持有，因此阻塞
- `非抢占`：只有资源的持有者才能释放它
- `循环等待`：==**形成了一个资源等待链条**==

## 如何处理死锁？

- 死锁预防
- 死锁避免
- 死锁检测和解除

## 死锁预防

用限制条件破坏四个条件的任意一个即可

## 死锁避免

>统筹规划资源的系统级算法

为描述释放资源的算法过程，定义如下的数据结构
- `系统`
	- `Available`：表示每种资源的可用数量
- `进程`
	- `Max`：每个进程对每种资源的最大需求量
	- `Allocation`：已经分配给每个进程的资源数量
	- `Need`：每个资源还需要的资源数量
$$
Need=Max-Allocation
$$

### 银行家算法

- 初始化，为每个进程和资源设置`Max`,`Allocation`和`Need`
- 当某进程请求资源，检查申请的资源是否小于等于`Need`
- 检查申请的资源量是否小于等于`Available`
- 模拟分配，将资源从`Available`中减去，加到进程的`Allocation`中，更新`Need`
- 安全性检查
- 如果安全性检查通过则执行资源分配，否则拒绝请求

### 安全分配序列

```ad-note
title:安全分配序列

安全分配序列算法，是为了找到一个进程的前后执行顺序，当进程按照这种顺序进行占用和释放资源时，不会产生死锁
```

先判断有没有进程可以安全执行，然后再找安全序列

### 安全性检查

算法需要维护以下数据：

- **Available（可用资源向量）**：系统当前可用的各类资源数量。
    
- **Max（最大需求矩阵）**：每个进程对各类资源的**最大**需求量。
    
- **Allocation（分配矩阵）**：当前已分配给每个进程的各类资源数量。
    
- **Need（需求矩阵）**：每个进程**尚需**的各类资源数量（Need = Max - Allocation）。

1. **初始化**：
    
    - 设置工作向量 `Work`，初始值等于 `Available`。
        
    - 设置完成向量 `Finish`，初始值全部为 `false`，表示所有进程都未完成。
        
2. **寻找可执行进程**：
    
    - 遍历所有进程，找到一个满足 `Finish[i] == false` 且 `Need[i] <= Work` 的进程 `P_i`。即该进程尚需的资源量不超过系统当前可用的资源量。
        
3. **模拟资源分配与回收**：
    
    - 如果找到这样的进程 `P_i`，假设它可以获得所需资源并运行至结束。
        
    - 然后它释放其占用的所有资源。因此，更新：`Work = Work + Allocation[i]`。
        
    - 标记该进程完成：`Finish[i] = true`。
        
    - 返回步骤2继续寻找。
        
4. **检查结果**：
    
    - 如果最终**所有** `Finish[i] == true`，则系统处于安全状态。刚刚模拟的进程顺序就是一个**安全序列**。
        
    - 如果存在 `Finish[i] == false` 的进程，则系统处于不安全状态。这意味着当前状态下，如果进程继续申请资源，可能引发死锁。        

## 死锁的检测和删除

```ad-warning
title:检测的是当前是否已经发生了死锁！
```

使用 **资源分配图**：

- 将系统中的所有资源和进程表示为图中的节点。
- 如果进程 P1 请求资源 R1，绘制从 P1 到 R1 的有向边。
- 如果资源 R1 分配给了进程 P2，绘制从 R1 到 P2 的有向边。

在构建完 **资源分配图**，可以通过使用 **DFS** 检查图中是否存在环路以判断是否存在**死锁**
